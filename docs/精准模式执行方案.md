# 精准模式执行方案（含代码落地路径）

## 1. 目标与范围

### 1.1 目标

在现有小程序中新增“精准模式（strict）”，并满足：

1. 到案（问卷）阶段可决定模式。
2. 用户可在档案页修改模式。
3. 模式会真实影响识别策略与提示词，不是“假开关”。
4. 识别链路（同步分析 + 异步任务）都能透传并执行该模式。

### 1.2 非目标（首版不做）

1. 不做复杂 AB 平台。
2. 不做自动化分群推荐引擎（先做规则推荐）。
3. 不在首版接入外部食堂/餐厅数据库。

---

## 2. 现状与可复用基础

当前项目已具备精准模式可复用的关键能力：

1. 健康档案写入链路完整：`GET/PUT /api/user/health-profile`。
2. 分析请求具备可扩展 payload：`AnalyzeRequest`、`AnalyzeSubmitRequest`。
3. 异步任务表 `analysis_tasks` 已有 `payload jsonb` 字段，可直接透传模式。
4. Worker 使用 `_build_food_prompt` 拼装提示词，适合插入“模式约束块”。

---

## 3. 一期架构设计

### 3.1 模式枚举

仅做两档，避免复杂化：

1. `standard`：标准模式（默认）。
2. `strict`：精准模式。

### 3.2 生效原则

1. 问卷提交时可写入模式。
2. 档案编辑时可修改模式。
3. 分析时优先使用请求体显式传入的 `execution_mode`。
4. 请求未传时，后端自动回退到用户档案里的 `execution_mode`。
5. 档案也未设置时，默认 `standard`。

## 4. 数据库改造（SQL）

在 `weapp_user` 增加执行模式字段，新增切换日志表。

```sql
-- 1) 用户主表新增字段
ALTER TABLE public.weapp_user
  ADD COLUMN IF NOT EXISTS execution_mode text DEFAULT 'standard'
    CHECK (execution_mode IN ('standard', 'strict')),
  ADD COLUMN IF NOT EXISTS mode_set_by text DEFAULT 'system'
    CHECK (mode_set_by IN ('system', 'user_manual', 'coach_manual')),
  ADD COLUMN IF NOT EXISTS mode_set_at timestamptz,
  ADD COLUMN IF NOT EXISTS mode_reason text,
  ADD COLUMN IF NOT EXISTS mode_commitment_days int DEFAULT 14,
  ADD COLUMN IF NOT EXISTS mode_switch_count_30d int DEFAULT 0;

COMMENT ON COLUMN public.weapp_user.execution_mode IS '执行模式: standard/strict';
COMMENT ON COLUMN public.weapp_user.mode_set_by IS '模式来源: system/user_manual/coach_manual';
COMMENT ON COLUMN public.weapp_user.mode_reason IS '模式变更原因编码';

-- 2) 模式切换日志
CREATE TABLE IF NOT EXISTS public.user_mode_switch_logs (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES public.weapp_user(id) ON DELETE CASCADE,
  from_mode text NOT NULL CHECK (from_mode IN ('standard', 'strict')),
  to_mode text NOT NULL CHECK (to_mode IN ('standard', 'strict')),
  changed_by text NOT NULL CHECK (changed_by IN ('system', 'user_manual', 'coach_manual')),
  reason_code text,
  created_at timestamptz DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_mode_switch_logs_user_id
  ON public.user_mode_switch_logs(user_id);
```

建议新建迁移文件：`backend/sql/add_execution_mode_to_user.sql`。

## 5. 后端改造（FastAPI）

## 5.1 请求模型扩展

文件：`backend/main.py`

1. `HealthProfileUpdateRequest` 增加字段：
   - `execution_mode: Optional[str]`
   - `mode_set_by: Optional[str]`
   - `mode_reason: Optional[str]`
2. `AnalyzeRequest`、`AnalyzeSubmitRequest` 增加：
   - `execution_mode: Optional[str]`

示例代码：

```python
class HealthProfileUpdateRequest(BaseModel):
    ...
    execution_mode: Optional[str] = Field(None, description="standard / strict")
    mode_set_by: Optional[str] = Field(None, description="system / user_manual / coach_manual")
    mode_reason: Optional[str] = Field(None, description="switch reason code")
```

## 5.2 档案接口返回字段

在以下接口返回中补充：

1. `GET /api/user/profile`
2. `GET /api/user/health-profile`
3. `PUT /api/user/health-profile` 返回体

新增返回字段：

1. `execution_mode`
2. `mode_set_by`
3. `mode_set_at`
4. `mode_reason`
5. `mode_commitment_days`
6. `mode_switch_count_30d`

## 5.3 PUT /api/user/health-profile 写入逻辑

写入逻辑建议：

1. 校验 `execution_mode in ('standard','strict')`。
2. 检测是否发生切换（与旧值不同）。
3. 若切换：
   - 写入 `mode_set_at=now()`
   - `mode_set_by` 默认 `user_manual`
   - `mode_switch_count_30d += 1`（首版可先简单累加）
   - 记录 `user_mode_switch_logs`

## 5.4 分析接口接入模式

接口：

1. `POST /api/analyze`
2. `POST /api/analyze/submit`
3. `POST /api/analyze-compare`

逻辑：

1. 优先读 `request.execution_mode`。
2. 无则读取用户 `execution_mode`。
3. 生成 `mode_hint` 注入 prompt。
4. 异步 submit 时把 `execution_mode` 写入 `payload`。

## 6. Worker 改造

文件：`backend/worker.py`

`_build_food_prompt(task, profile_block)` 里读取：

```python
execution_mode = (payload.get("execution_mode") or "standard").strip().lower()
```

加入模式提示块：

```python
def _build_mode_hint(execution_mode: str) -> str:
    if execution_mode == "strict":
        return """
执行模式：精准模式（strict）
- 仅识别单纯碳水或单纯瘦肉。
- 若为混合食物、重油烹饪、肥瘦不明，请明确提示“需分开拍/拨开拍/重拍”。
- 低置信度时不要给确定克数，给出重拍建议。
"""
    return """
执行模式：标准模式（standard）
- 可提供估算值，但需在不确定时给范围并提示偏差风险。
"""
```

然后拼进 prompt 中，确保同步和异步分析行为一致。

## 8. 规则与文案（产品必做）

## 8.1 精准模式用户提示（固定文案）

1. 仅支持单纯碳水、单纯瘦肉识别。
2. 混合食物需分开拍。
3. 低置信度会要求重拍，不强行给克数。

## 8.2 模式切换确认文案

从 `strict -> standard`：

“切换到标准模式后记录更便捷，但克数准确性会下降，可能影响减脂/增肌进度。”

从 `standard -> strict`：

“精准模式会增加拍照规范要求，但能提升计划执行可靠性。”

## 11. 回归用例清单（最小集）

1. 新用户问卷选择 `strict`，提交后档案读回一致。
2. 已有用户从 `standard` 切到 `strict`，日志表新增记录。
3. `submitAnalyzeTask` 未传 `execution_mode` 时，后端自动使用档案值。
4. `strict` 下上传混合菜图，返回重拍/分开拍提示，不给伪精确克数。
5. `standard` 下同图可给估算值，并带偏差提示。
6. 异步任务（worker）与同步分析（/api/analyze）行为一致。

## 12. 一期验收标准（Definition of Done）

1. 用户可在“到案”和“档案编辑”两处配置执行模式。
2. 模式字段在数据库、接口、前端状态三层一致。
3. 识别 prompt 已根据模式产生差异化行为。
4. 至少 6 条回归用例全部通过。
5. 指标埋点可在日志或数据平台中查到。
