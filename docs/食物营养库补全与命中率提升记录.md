# 食物营养库补全与命中率提升记录

## 背景

本轮工作的目标是把“拍照识别食物热量”从依赖大模型直接给出热量，逐步切换为：

1. 大模型只负责识别食物名称和重量。
2. 后端优先从 `food_nutrition_library` 查询每 100g 营养数据。
3. 再根据重量换算出总热量和营养成分。
4. 对历史高频未命中的食物，持续补充 `food_nutrition_library` 和 `food_nutrition_aliases`。

这轮工作同时覆盖了：

- 食物营养字段扩展
- USDA 数据导入
- USDA 中文 alias 生成
- 历史命中率统计
- 历史缺失食物自动补库
- 数据来源字段规范化

### 6. 从历史记录自动补库

编写了历史缺失食物补库脚本，按以下顺序补：

1. 先复用现有营养库，补 alias
2. 再用历史记录中的重量和营养数据，反推每 100g 平均值，补 `food_nutrition_library`
3. 再用 AI 估计每 100g 营养值，作为兜底
4. 最后才尝试网页搜索类方案

涉及文件：

- [backend/scripts/enrich_top_missing_foods.py](/K:/code/jinhui/food-link/food_link/backend/scripts/enrich_top_missing_foods.py)

这个脚本后来又继续做了以下增强：

- 支持按 `user_food_records.items` 统计真实历史缺失
- 先过滤已覆盖食物，再取 top N
- 组合句拆分
- 数量和单位剥离
- 低质量候选过滤
- 历史均值补库
- AI 保底补库
- 插入幂等化，避免“第一次成功、回包失败、第二次撞唯一键”导致任务中断
- 把默认参数调整为“尽可能全量补历史缺失”

把 `food_nutrition_library.source` 统一为“营养数据真实来源”，不再记录内部规则过程名。

来源示例：

- 历史识别记录营养均值
- AI估算（DeepSeek）

## 哪些文件不建议提交

不建议提交的主要是“运行脚本生成的中间文件或导出产物”。

例如：

- `backend/database/usda_preview.json`
- `backend/database/usda_alias_preview.json`
- `backend/database/top_missing_preview.json`
- `backend/database/top_missing_import.json`
- `backend/database/top_missing_import_full.json`
- `backend/database/top_missing_ai_debug.json`
- `backend/database/usda_import.sql`
- `backend/database/usda_alias_preview.sql`

原因：

- 这些文件可以由 Python 脚本重新生成
- 文件通常很大，改动噪音很重
- 容易让 commit 里混入大量无业务价值的导出内容
- 一旦数据重新导出，diff 很难读

原则上：

- “逻辑代码”和“迁移脚本”提交
- “脚本执行结果”和“导出快照”不提交

## 当前经验结论

1. 命中率的大头提升，来自“历史高频缺失补库”。
2. `food_nutrition_library` 和 `food_nutrition_aliases` 需要同时补，效果才明显。
3. 历史均值回填对高频食物很有效。
4. AI 兜底适合补最后一层长尾缺失。
5. 搜索引擎抓网页方案不稳定，只适合作为补充，不适合作为主流程。
6. 生成产物文件不适合提交到仓库。
