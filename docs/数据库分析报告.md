# Food Link 数据库分析报告

## 说明

本次分析目标原本是通过 Supabase MCP 直接读取数据库现状，但当前工作区未发现可用的 `supabase` MCP 服务描述或连接入口，因此本报告基于以下材料整理：

- `backend/database/*.sql`
- `backend/sql/*.sql`
- `backend/database.py`
- `backend/main.py`
- `backend/worker.py`

这意味着：本报告能较准确反映“项目设计中的数据库结构”和“代码实际依赖的数据库能力”，但不能确认线上 Supabase 实库的真实表数量、字段完整性、索引是否全部执行、数据量、RLS 策略和实际存储分布。

## 一、数据库整体定位

这是一个围绕“饮食记录 + AI 分析 + 社区互动 + 健康档案”的业务数据库，核心数据库职责大致分为 6 层：

1. 用户与健康档案
2. 饮食记录与分析任务
3. 公共食物库
4. 社交关系与圈子互动
5. 内容审核与违规治理
6. AI 运营支撑数据

整体上看，库设计偏业务驱动，表结构直接服务 API 和 Worker 流程，很多能力通过 `jsonb` 做柔性扩展，适合快速迭代，但也带来一定的结构漂移风险。

## 二、核心表分层

### 1. 用户与健康档案

#### `weapp_user`

这是全库主用户表，承担账号、基础资料、健康档案、隐私配置、执行模式配置等多重职责。

主要字段分组：

- 身份字段：`id`、`openid`、`unionid`
- 资料字段：`avatar`、`nickname`、`telephone`
- 健康字段：`height`、`weight`、`birthday`、`gender`、`activity_level`
- 代谢字段：`bmr`、`tdee`
- 健康档案扩展：`health_condition`（`jsonb`）
- 引导状态：`onboarding_completed`
- 目标与策略：`diet_goal`
- 精准模式相关：`execution_mode`、`mode_set_by`、`mode_set_at`、`mode_reason`、`mode_commitment_days`、`mode_switch_count_30d`
- 隐私设置：`searchable`、`public_records`

结论：

- `weapp_user` 已明显演化成“用户主档 + 健康画像 + 隐私配置 + 策略配置”聚合表。
- 这种做法开发效率高，但字段会持续膨胀，后续建议逐步拆分用户账号域与健康画像域。

#### `user_health_documents`

用户上传的健康报告 / 体检报告 OCR 结果表，用于保存图片 URL 和解析后的 `jsonb` 内容。

用途：

- 健康报告 OCR 导入
- 给 AI 生成个性化建议时提供结构化背景

#### `user_mode_switch_logs`

记录标准模式和精准模式之间的切换日志，适合做策略分析和产品埋点。

### 2. 饮食记录与分析任务

#### `user_food_records`

这是业务最核心的事实表，存用户每一餐的记录。

主要字段：

- 归属与时间：`user_id`、`record_time`、`created_at`
- 餐次：`meal_type`
- 图片：`image_path`
- 文本结果：`description`、`insight`
- 结构化结果：`items`（`jsonb`）
- 汇总营养：`total_calories`、`total_protein`、`total_carbs`、`total_fat`、`total_weight_grams`
- 深度分析扩展：`pfc_ratio_comment`、`absorption_notes`、`context_advice`
- 场景字段：`diet_goal`、`activity_timing`
- 兼容旧版：`context_state`
- 来源追踪：`source_task_id`

特征判断：

- 这是典型的“结果快照表”，AI 识别结果在落库时就已经聚合完成。
- `items jsonb` 是核心设计，说明系统优先服务快速展示和低成本迭代，而不是做高度规范化营养明细建模。

#### `analysis_tasks`

异步分析任务表，由 API 创建、Worker 抢占执行。

主要支持：

- 图片分析：`task_type='food'`
- 文字分析：`task_type='food_text'`
- 健康报告分析：`task_type='health_report'`
- 公共食物库文案审核：`task_type='public_food_library_text'`

状态流转：

- `pending`
- `processing`
- `done`
- `failed`
- `violated`

扩展字段：

- 输入：`image_url`、`text_input`、`payload`
- 输出：`result`
- 审核：`is_violated`、`violation_reason`

结论：

- `analysis_tasks` 是整个 AI 工作流的中枢表。
- `user_food_records.source_task_id` 把“分析任务”和“最终饮食记录”串起来了，这个设计是合理的。

#### `ai_stats_insights`

这是统计页 AI 洞察缓存表，避免同一天同一范围重复调用大模型。

唯一键：

- `(user_id, range_type, generated_date)`

这个设计清晰，适合缓存“周报/月报”类文案。

#### `critical_samples_weapp`

用于收集 AI 重量估算偏差样本，属于模型反馈闭环数据。

### 3. 公共食物库

#### `public_food_library`

这是“用户记录 -> 社区分享内容”的主表。

主要字段：

- 来源：`user_id`、`source_record_id`
- 内容：`image_path`、`image_paths`、`items`、`description`、`insight`
- 营养汇总：`total_calories`、`total_protein`、`total_carbs`、`total_fat`
- 商户与地理：`merchant_name`、`merchant_address`、`detail_address`、`province`、`city`、`district`、`latitude`、`longitude`
- 社区属性：`food_name`、`taste_rating`、`suitable_for_fat_loss`、`user_tags`、`user_notes`
- 审核与发布：`status`、`audit_reject_reason`、`published_at`
- 计数：`like_count`、`comment_count`、`avg_rating`、`collection_count`

状态值：

- `pending`
- `published`
- `rejected`

这是一个典型的“审核后发布”的 UGC 内容表。

#### `public_food_library_comments`

公共食物库评论表，支持评分 `rating`。

#### `public_food_library_collections`

公共食物库收藏表，已通过触发器维护 `public_food_library.collection_count`。

### 4. 社交关系与圈子互动

#### `user_friends`

好友关系表，设计为双方各插一条记录，方便单向查询。

#### `friend_requests`

好友申请表，状态为：

- `pending`
- `accepted`
- `rejected`

#### `feed_likes`

针对 `user_food_records` 的点赞表，唯一键 `(user_id, record_id)`。

#### `feed_comments`

针对 `user_food_records` 的评论表。

说明：

- 圈子 Feed 的内容本质上不是独立发帖表，而是直接复用 `user_food_records`。
- 这意味着“记录事实”和“社交内容”绑定得很紧，优点是简单，缺点是后续做更复杂社区能力时受限。

### 5. 内容审核与违规治理

#### `comment_tasks`

评论异步审核任务表，审核通过后再落真实评论表。

支持目标：

- `feed`
- `public_food_library`

#### `content_violations`

违规内容留痕表，记录违规分类、原因、图片、文本、关联记录等。

用途：

- AI 审核留档
- 风控追踪
- 后续运营复盘

### 6. AI 运营支撑

#### `model_prompts`

模型提示词配置表，支持按模型类型管理激活提示词。

#### `model_prompts_history`

提示词修改历史表。

这部分说明项目已经把“模型行为配置”也纳入数据库管理，而不是只写死在代码里。

## 三、主要关系图

核心关系可以概括为：

- `weapp_user` 1:N `user_food_records`
- `weapp_user` 1:N `analysis_tasks`
- `analysis_tasks` 1:N `user_food_records`（逻辑上通常 1:1 或 1:N，通过 `source_task_id` 关联）
- `weapp_user` 1:N `public_food_library`
- `user_food_records` 1:N `feed_comments`
- `user_food_records` 1:N `feed_likes`
- `public_food_library` 1:N `public_food_library_comments`
- `public_food_library` 1:N `public_food_library_collections`
- `weapp_user` N:N `weapp_user`，通过 `user_friends`

从业务链路看，最关键的一条主链是：

`weapp_user` -> `analysis_tasks` -> `user_food_records` -> `public_food_library`

## 四、典型业务流

### 1. 饮食拍照识别

1. API 创建 `analysis_tasks`
2. Worker 抢占任务并调用模型
3. 结果写回 `analysis_tasks.result`
4. 用户确认后落 `user_food_records`
5. 通过 `source_task_id` 保留来源链路

### 2. 体检报告导入

1. 图片上传 Supabase Storage `health-reports`
2. 识别结果写入 `user_health_documents`
3. 同步更新 `weapp_user.health_condition`

### 3. 公共食物库分享

1. 来源可能是已有 `user_food_records`
2. 创建 `public_food_library`，初始状态 `pending`
3. 审核任务以 `public_food_library_text` 执行
4. Worker 更新为 `published` 或 `rejected`

### 4. 评论审核

1. 用户发表评论先写 `comment_tasks`
2. Worker 审核
3. 通过后落 `feed_comments` 或 `public_food_library_comments`
4. 违规时写 `content_violations`

## 五、当前设计优点

### 1. 业务闭环完整

从用户录入、AI 处理、结果沉淀、社区分享、内容审核到模型反馈样本，链路基本闭环。

### 2. 对异步 AI 任务支持较好

`analysis_tasks` 和 `comment_tasks` 两类任务表把 API 请求与 Worker 执行解耦了，结构上比较清晰。

### 3. 扩展速度快

大量使用 `jsonb`，对快速迭代友好，尤其适合 AI 返回结构和健康档案这类高变化字段。

### 4. 索引意识较强

大部分高频查询字段已经有索引，比如：

- 用户维度
- 时间维度
- 状态维度
- 部分榜单和筛选字段

## 六、主要问题与风险

### 1. 代码依赖与迁移脚本存在不一致

最明显的问题有：

- 代码在读写 `public_food_library_likes`，但当前仓库内未找到该表的建表 SQL
- `database.py` 使用了 `analysis_tasks.image_paths` 和 `user_food_records.image_paths`，但这两个字段只出现在单独迁移脚本中，不在基础建表 SQL 里
- `weapp_user` 的 `searchable`、`public_records`、`execution_mode` 等字段也依赖额外迁移，不在最初主表定义里

影响：

- 新环境初始化很容易“代码已上线、字段没补齐”
- 本地、测试、线上三个环境可能结构不一致

### 2. 主表字段膨胀明显

`weapp_user` 和 `user_food_records` 都承载了越来越多横向能力：

- 用户主档
- 健康档案
- 隐私设置
- 执行策略
- 场景分析字段

影响：

- 后续维护成本会越来越高
- 接口更新一个点，可能牵动多个不相关业务字段

### 3. 过度依赖 `jsonb`

以下字段都是 `jsonb` 核心承载：

- `weapp_user.health_condition`
- `user_food_records.items`
- `analysis_tasks.payload`
- `analysis_tasks.result`
- `user_health_documents.extracted_content`
- `public_food_library.items`

问题不是不能用，而是如果未来要做：

- 更复杂统计
- 标签/食材级搜索
- 精细推荐
- 运营筛选

那查询成本和索引设计会变难。

### 4. 公共食物库计数字段维护机制不完整

目前明确看到有触发器维护的是：

- `collection_count`

但以下字段虽在主表中存在，暂未在现有 SQL 中看到对应维护机制：

- `like_count`
- `comment_count`
- `avg_rating`

如果这些值靠代码手动维护，容易出现计数不一致。

### 5. 社区模型偏“记录复用”，不利于后续内容产品化

当前圈子 Feed 直接复用 `user_food_records`，这很轻，但也意味着：

- 无独立发帖对象
- 无正文编辑能力
- 无更细粒度可见性控制
- 难支持后续转发、合集、主题贴等能力

### 6. 迁移文件分散，缺少统一基线

当前数据库结构由多份文件叠加形成：

- 基础建表 SQL
- 多个增量迁移 SQL
- 一些“可选迁移”脚本

但没有看到一个明确的“从 0 到最新”的基线脚本或迁移管理规范。

影响：

- 新环境初始化依赖人工顺序执行
- 很容易漏脚本
- 排查线上字段缺失会更慢

## 七、结构成熟度判断

如果从产品阶段来判断，这套数据库更像：

- 已脱离 MVP 初版
- 正处于“功能扩张期”
- 还没有进入“统一收敛和治理期”

它已经能支撑当前产品功能，但数据库治理还没完全跟上业务扩张速度。

## 八、建议的优化优先级

### P0：先解决结构一致性

建议尽快补齐一份“当前线上应有的完整 schema 基线”，至少覆盖：

- 所有主表
- 所有外键
- 所有状态约束
- 所有当前代码依赖字段
- `public_food_library_likes` 这类缺失定义的表

### P1：统一迁移管理

建议把散落 SQL 收敛成可追踪迁移序列，例如：

- `001_init.sql`
- `002_analysis_tasks_text.sql`
- `003_user_food_records_extensions.sql`

目标是让新环境可一次性恢复到最新状态。

### P1：梳理公共食物库计数真相

确认以下字段究竟是：

- 强一致实时维护
- 定时回刷
- 仅展示层动态计算

并统一实现方式：

- `like_count`
- `comment_count`
- `avg_rating`

### P2：逐步拆分用户域

长期建议把 `weapp_user` 拆成：

- 用户账号主表
- 用户健康画像表
- 用户隐私配置表
- 用户执行策略表

这样能降低主表膨胀和联动修改风险。

### P2：为 JSONB 建“稳定投影字段”

对未来高频筛选统计项，不要只存在 `jsonb` 内，建议补充稳定列或物化表。

典型候选包括：

- 饮食记录中的主要食物标签
- 健康档案中的关键目标/禁忌/过敏原
- 公共食物库中的结构化筛选项

## 九、结论

这套数据库不是一个简单的“用户 + 记录”库，而是已经发展成一个面向 AI 饮食产品的复合型业务库。它的强项是业务闭环完整、AI 任务解耦清晰、迭代速度快；它的主要隐患是迁移管理偏散、代码与 schema 有漂移、主表膨胀和 JSONB 依赖较重。

如果你现在要把它继续往线上规模化推进，最优先不是再加表，而是先做一次数据库结构收口，把“当前真实可运行 schema”固定下来。
