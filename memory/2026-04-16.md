# 2026-04-16

## 修复真机调试切换日期/分享餐食报 `c is not a function`

- **根因**：Taro 4.x 的 Vite runner 在小程序端硬编码 `build.target: 'es6'`，导致 `async/await` 被编译为 generator + 内联辅助函数 `c=(e,t,s)=>new Promise(...)`。该短变量名辅助函数与代码中局部变量冲突，在微信小程序真机 JSCore 上偶发 `c is not a function`。
- **修复**：在 `config/index.ts` 的 `compiler.vitePlugins` 中添加 `taro-fix-target` 插件，在 `configResolved` 阶段将 `build.target` 覆盖为 `es2018`，保留原生 `async/await`，彻底消除内联 generator 辅助函数。
- **验证**：
  - `npm run build:weapp` 产物中 `function*` 和 `yield` 已清零，`async/await` 被保留。
  - 重启前后端（`npm run dev:restart`），开发者工具自动化验证：切换日期正常、分享海报正常、无控制台报错。
  - 待真机扫码复验确认问题消失。

## 修复首页日期热图颜色全白 + 本地 storage 自动补齐 7 天

- **根因**：`loadDashboard` 与 `syncDashboardForDate` 共用同一个 `loadDashboardSeqRef`，且 `loadDashboardPendingRef` 的并发跳过逻辑被放在 `++seq` 之后。当 `useDidShow` 等场景快速触发时，第二个调用先增加了 `seq` 然后被 pending 拦截，导致第一个正在进行的 `loadDashboard` 在 `Promise.all` 完成后因 `seq` 不匹配而被整体丢弃，`weekHeatmapCells` 永远没被写入状态。
- **修复**：
  1. `loadDashboard` 中将 `loadDashboardPendingRef` 同日期跳过判断**提前到 `++seq` 之前**，避免无效调用污染 seq。
  2. `syncDashboardForDate` 改用**独立的 `syncDashboardSeqRef`**，不再与 `loadDashboard` 共用 seq，彻底消除两者之间的竞态丢弃。
- **storage 补齐**：在 `loadDashboard` 成功后，若 `getStoredHomeDashboardSnapshots().length < 7`，则后台并发请求缺失的近 7 天 `getHomeDashboard` 数据并写入 `home_dashboard_local_cache_v3`；补齐完成后从 storage 重新读取当前选中日期数据并刷新 UI。
- **验证**：
  - 开发者工具 DOM 检查：13-16 号圆圈已正确显示 `is-recorded`（绿色），17-19 号无记录保持 `is-empty`（白色）。
  - storage 检查：本地缓存已从 1 条自动补齐到 7 条（2026-04-10 ~ 04-16）。
