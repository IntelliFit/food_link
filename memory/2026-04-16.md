## 修复真机调试切换日期/分享餐食报 `c is not a function`

- **根因**：Taro 4.x 的 Vite runner 在小程序端硬编码 `build.target: 'es6'`，导致 `async/await` 被编译为 generator + 内联辅助函数 `c=(e,t,s)=>new Promise(...)`。该短变量名辅助函数与代码中局部变量冲突，在微信小程序真机 JSCore 上偶发 `c is not a function`。
- **修复**：在 `config/index.ts` 的 `compiler.vitePlugins` 中添加 `taro-fix-target` 插件，在 `configResolved` 阶段将 `build.target` 覆盖为 `es2018`，保留原生 `async/await`，彻底消除内联 generator 辅助函数。
- **验证**：
  - `npm run build:weapp` 产物中 `function*` 和 `yield` 已清零，`async/await` 被保留。
  - 重启前后端（`npm run dev:restart`），开发者工具自动化验证：切换日期正常、分享海报正常、无控制台报错。
  - 待真机扫码复验确认问题消失。

## 修复首页日期热图颜色全白 + 本地 storage 自动补齐 7 天

- **根因**：`loadDashboard` 与 `syncDashboardForDate` 共用同一个 `loadDashboardSeqRef`，且 `loadDashboardPendingRef` 的并发跳过逻辑被放在 `++seq` 之后。当 `useDidShow` 等场景快速触发时，第二个调用先增加了 `seq` 然后被 pending 拦截，导致第一个正在进行的 `loadDashboard` 在 `Promise.all` 完成后因 `seq` 不匹配而被整体丢弃，`weekHeatmapCells` 永远没被写入状态。
- **修复**：
  1. `loadDashboard` 中将 `loadDashboardPendingRef` 同日期跳过判断**提前到 `++seq` 之前**，避免无效调用污染 seq。
  2. `syncDashboardForDate` 改用**独立的 `syncDashboardSeqRef`**，不再与 `loadDashboard` 共用 seq，彻底消除两者之间的竞态丢弃。
- **storage 补齐**：在 `loadDashboard` 成功后，若 `getStoredHomeDashboardSnapshots().length < 7`，则后台并发请求缺失的近 7 天 `getHomeDashboard` 数据并写入 `home_dashboard_local_cache_v3`；补齐完成后从 storage 重新读取当前选中日期数据并刷新 UI。
- **验证**：
  - 开发者工具 DOM 检查：13-16 号圆圈已正确显示 `is-recorded`（绿色），17-19 号无记录保持 `is-empty`（白色）。
  - storage 检查：本地缓存已从 1 条自动补齐到 7 条（2026-04-10 ~ 04-16）。

## 首页修改食物参数优化：消除 loading + 支持编辑餐次/饮食目标/运动时机

- **问题 2：无法修改餐次、饮食目标、运动时机**

- **真机日期切换 React #130 崩溃修复**
  - **根因**：`full_record`（完整饮食记录，含所有食物明细、营养值、图片路径等）随 dashboard 返回后直接进入 React `meals` state。在微信小程序真机上，React state 会通过 `setData` 深序列化后传递到 WXML 层。连续快速切换日期时，多次大对象深序列化/反序列化可能导致 JS-Native 桥数据损坏，进而触发 React reconciler 的 `Minified React error #130`（Element type is invalid: got object）。
  - **修复**：将 `full_record` 从 React state 中彻底剥离，改由 `api.ts` 的模块级 `mealFullRecordCache` 承载。首页 `meals` 数据不再包含 `full_record`，但点击「修改记录」时仍能通过 `getCachedMealFullRecord` 实现零 loading 秒开。

## 真机日期切换崩溃问题 —— 排查进展

- **现象**：`npm run build:weapp:debug`（development，无 Terser 压缩）真机正常；`npm run build:weapp`（production，Terser 压缩）真机崩溃。
- **真机日志关键线索**：
  1. `TypeError: Assignment to constant variable.`（未处理的 Promise rejection）
  2. 发生在 `loadDashboard` 成功后、后台补齐近 7 天缓存的并发请求期间
  3. 随后日期切换触发 `React error #130`
- **根因判断**：Terser 的 production 压缩产物在微信小程序真机 JSCore 上触发了语法/语义级 bug。development 构建无 Terser 压缩，因此完全正常。
- **已做修复**：
  1. `config/prod.ts`：Terser 配置禁用 `mangle`（变量名不再混淆），保留基础 `compress`。
  2. `src/pages/index/index.tsx`：后台补齐 IIFE 中的 `for (let offset=-6; ...)` 改为 `Array.from({length:7}).forEach(...)`，规避 Terser 对带闭包 `for` 循环的已知优化 bug；并给整个 IIFE 包裹 `try-catch`，避免未处理 rejection。
  3. 清理 Vite 缓存后重新构建：`rm -rf node_modules/.vite dist && npm run build:weapp`。
- **下一步**：用新产物在真机上测试，观察是否还出现 `Assignment to constant variable` 和 `React error #130`。若仍复现，则进一步禁用 Terser `compress`。
